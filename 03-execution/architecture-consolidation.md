# Architecture Consolidation: From Fleet-of-One to Multi-Tenant V1

## Headline

Nothing in the current estate is multi-tenant-shaped, but three of the four pillars (OKR scoring, meeting capture, gamification math) are real, working code worth carrying forward as libraries, not rewrites. The fourth (front-door UI) has to be rebuilt from scratch, though the patterns it proved are reusable. The native-vs-web tension resolves cleanly once you separate "operator console" from "client product": the estate's native law was proven at n=1 for an internal fleet HUD, not for external distribution, and current enterprise procurement (SSO/SAML as a deal-blocker) plus the fact that OAuth in an embedded native webview is itself against spec (RFC 8252) means native buys nothing extra on the auth axis. ClickUp should stay the backbone for now because of sunk investment and because the outcome-scoring gap is explicitly blocked on it, but it must be integrated through an adapter and restricted to the public API plane only; the internal-frontdoor hack that Jarred's own capability map already flags as ToS-gray and suspension-risk cannot be shipped against client-owned workspaces. Linear is the more agent-native backbone as of July 2026 (it is a named adopter of MCP's new stable enterprise auth extension; ClickUp is not), which argues for building the adapter layer with Linear as the second target, not an afterthought. The Claude Agent SDK and MCP give the product a real, free head start on tool auth and model access, but they explicitly do not give multi-tenancy, budget caps, retries, or observability; those are exactly the gaps the estate has already identified on its own (pai-observatory, teamclaude-fleet) and they remain the actual engineering lift for V1.

## Findings

### 1. What the estate actually is, on disk (repo-reality check, this session)

- `claudecode-bar` (322 commits, most active repo in the estate): a native macOS app, `main.swift` at the root, packaged as `claudecode-bar.app`. It renders its panels (`hud.html`, `console.html`, `dashboard.html`, `recall.html`, `browser-live.html`) as embedded web views inside the native shell, driven by Python bridge scripts and Chrome DevTools Protocol automation, not a from-scratch native UI toolkit build. This is the concrete shape of "native app" the product-law rule (#13 in the concept overview, "native-app delivery for V1 surfaces") is actually describing: a native chrome around web guts, single-user, single-machine, no auth or tenancy anywhere in the codebase.
- `the-sage` (58 commits): a Python package (`pyproject.toml`, `src/`) implementing the OKR engine: `goals.json` schema and `score_alignment()`, validated at 99.3 percent on 275 of 277 real commits. This is real, working scoring logic, not a prototype.
- `teamclaude-fleet`: Node/TypeScript (`package.json`, `src/`), the multi-account OAuth quota rotation layer the bar's cost odometer consumes. This is closer to infra plumbing than product surface, but it is the one piece of the estate already built with a service/API shape rather than a single-user script.
- `executive-agent-gui` and `command-center`: no application code. `executive-agent-gui` has only `INTENT.md`, `LIVENESS.md`, `README.md`, `memory-bank/`, `research/`, `scripts/`; `command-center` has a `bridge.py` and `omnara_cdp.py` (both automation glue) plus a `deck.html`, not a running product. Both match the overview's own read: designed/specced or tabled, not shipping.
- The ClickUp integration is the deepest and most mature integration in the whole estate (see `research/synthesis/clickup-capability-map.md`, 2026-06-18): a documented public REST/v3 plane (Goals/KeyResults CRUD, webhooks with HMAC verification, docs, comments, views, sprints) plus a second, undocumented "frontdoor" plane reached by harvesting a workspace JWT out of an authenticated browser tab and replaying it with a captured 6-header set. The map's own words: "ToS-gray, fragile, can break w/o notice," "genuine account-suspension risk automating an internal API," and "scope to the owned workspace." That last phrase is the load-bearing one for a multi-tenant product: it was written assuming one workspace (Jarred's own, team `8670502`), not N client workspaces.

### 2. Native app vs web app for enterprise distribution

- SSO is now a procurement gate, not a feature. Multiple 2026 B2B SaaS guides describe SAML/OIDC support as a hard blocker on mid-market and enterprise deals, with the first $30-50k contract typically arriving with a security questionnaire that requires it (WorkOS, ScaleKit, SSOJet, 2026 guides; see Sources).
- Enterprises expect SAML and OIDC both, plus SCIM lifecycle provisioning, JIT provisioning, role mapping, and audit logs; "most SSO implementations fail not because of login, but because of everything around it" (ScaleKit / SSOJet, 2026).
- Native app guidance in 2026 explicitly recommends web delivery for exactly this product's shape: "web apps are recommended for... CRM, HRM, data management software, monitoring software, project management and task tracking systems" (AWS / Lovable / Hostinger comparisons, 2026), while native is argued for when the app needs camera, GPS, biometrics, Bluetooth, or reliable offline use, none of which describe a chat-plus-triage-cards front door.
- The reconciling technical fact: OAuth inside an embedded native webview is a documented anti-pattern under RFC 8252 (the OAuth 2.0 spec written specifically for native apps), and this is true in both major native-shell frameworks in 2026. Native apps that want SSO must open the system browser and do PKCE with a loopback or deep-link callback; an embedded webview also cannot share the OS/browser SSO session, so users would have to re-authenticate every time, which is worse UX than a plain web app in a browser tab that is often already SSO'd in.
- Net effect: "native" does not buy you a better SSO story. It buys you OS-level chrome (menu bar presence, always-on HUD, local process click-to-debug) which is exactly what `claudecode-bar` uses it for today, and which is a legitimate operator-console argument, not a client-distribution argument.
- If a native shell is still wanted (for calm/focus branding or an operator console), the 2026 framework consensus favors Tauri for new builds: roughly 10x smaller binaries, 50-75 percent less memory, and a default-locked-down capability model that fits the estate's own "never auto-act without a gate" law at the OS permission layer. The trade-off: Electron has the more mature enterprise-SSO tooling today (MSAL Node for Entra ID, a path toward Windows Authentication Broker/WAM for device-bound Conditional Access), while Tauri's equivalent requires custom Rust plugin work per the Tauri maintainers themselves. If an enterprise buyer (for example, a bank-grade compliance team) demands device-level Conditional Access, that is a concrete point in Electron's favor; otherwise Tauri is the safer new-build default.

The one genuinely new July 2026 data point worth weighing heavily: Linear, not ClickUp, is a listed server-side adopter of MCP's Enterprise-Managed Authorization extension, which is the exact mechanism ("sign in once through your IdP, no per-server consent prompts") a multi-tenant agent product needs for its tool-connector layer. That is a real, current signal about which vendor is investing in agent-native enterprise auth, independent of either company's general project-management merits.

### 4. Claude Agent SDK and MCP as substrate

- The Agent SDK (renamed from the Claude Code SDK in September 2025) is open source and free to install; cost comes entirely from token consumption, billed against either a Claude subscription or API credits. It ships subagents, sessions, MCP tool-calling, and a bundled CLI.
- What it explicitly does NOT give you, per a 2026 production-patterns guide and a separate SDK guide: no built-in retry logic, no default turn/budget limits (`max_turns` defaults to unlimited), no durable execution, no multi-tenant deployment model, and no centralized observability. These are exactly the gaps the estate has already flagged for itself (the `pai-observatory` scaffold, the `teamclaude-fleet` quota/cost layer) so the existing instinct to build those pieces is correct; they are the real, necessary V1 engineering lift, not wasted motion.
- Billing risk, current as of this research: Anthropic announced on May 13, 2026 that programmatic/Agent-SDK usage would move off the shared subscription pool onto a separate metered credit pool at full API rates, effective June 15, 2026. That change is currently paused per Anthropic's own help center, with Anthropic stating it is "revising the plan" and will give advance notice before any future change. This is a live, unresolved platform risk: a multi-tenant product whose backend runs thousands of tenant-agent sessions through this SDK should build pricing headroom now for the scenario where Anthropic reinstates full API-rate metering with no subscription discount, since they have already announced intent to do exactly that once.
- MCP itself crossed into "default integration layer" territory in 2026: the protocol's next spec (finalizing July 28, 2026) makes the core stateless, addressing a named enterprise-scaling pain point, and the Enterprise-Managed Authorization extension is now stable, adopted by Anthropic, Microsoft, and Okta, with growing server-side support. Adoption is real but uneven: Stacklok's 2026 report puts 41 percent of surveyed software organizations in limited-or-broad MCP production (a correction of an earlier, unsourced 78 percent claim), Anthropic cited 10,000+ active public servers and 97M+ monthly SDK downloads as of December 2025, and a May 2026 registry pull counted 9,652 servers. Security has not kept pace: 30+ CVEs were filed in January-February 2026, and named production incidents include an Asana cross-tenant data leak and a Smithery path-traversal bug exposing 3,243 apps. Only about 8.5 percent of MCP servers currently implement the mandatory OAuth 2.1 standard for remote deployments, which is the direct cautionary tale for the product's own tool-connector layer: treat "it's MCP" as necessary, not sufficient, for a secure client-facing integration.
- On the sovereignty worry Carter raised ("if this runs on Anthropic, they'll just replicate what we do"): the integration and governance layer is now genuinely vendor-neutral. In December 2025, Anthropic, Block, and OpenAI placed MCP and the related A2A protocol under a new Agentic AI Foundation inside the Linux Foundation. That means the tool-connector and auth layer the product builds on MCP is portable to a different model provider later if needed. It does not change the fact that the actual reasoning/agent runtime (the Agent SDK, the model weights) remains Anthropic-specific today; there is no cross-provider abstraction in the SDK itself. The honest framing for the deck: the plumbing is de-risked, the brain is not.

## Options considered

1. **Full rebuild of all four pillars for multi-tenancy.** Rejected as the default. Throws away validated logic (the-sage's 99.3-percent-accurate scoring, the capture pipeline's 343 processed notes and 47k embeddings that the overview itself flags as "do NOT rebuild") for the sake of architectural cleanliness. Only justified for the front-door UI, where nothing shipping actually exists to consolidate.
2. **Port claudecode-bar directly into the client-facing product.** Rejected. It is a native, single-user, single-machine Swift app with zero auth or tenancy model; porting it means building tenancy and identity into a codebase that was never designed for either, while also fighting the native-distribution friction (app-store review, separate Windows build, no natural SSO story) documented above. Its patterns (two-pane layout, Approve/Edit/Skip cards, the fleet-HUD operator view) should migrate; its code should not.
3. **Consolidate the-sage, the capture pipeline, and the gamification math as internal libraries behind a new multi-tenant service layer; rebuild only the front door.** Recommended (below).
4. **Switch the backbone to Linear outright.** Considered given Linear's cleaner API reputation and its MCP enterprise-auth adoption, but rejected as a wholesale swap because it discards the estate's single deepest integration (ClickUp Goals/KeyResults, the outcome-scoring data source, the ops-tracker) for a vendor with no current sunk work. An adapter that supports both, defaulting to ClickUp for continuity and Linear for new client environments that are already Linear shops, captures the upside of both without the rewrite cost.
5. **Ship the client product as a native (Electron or Tauri) desktop app to honor the "native app" product law.** Considered and narrowed, not rejected outright: keep native for the internal operator console (where it already earns its keep), but treat the client-facing multi-tenant V1 as a responsive web app, since the actual argument for native (OS-level always-on presence, local process control) does not apply to what a client employee needs (chat, triage cards, decision cards).

## Recommendation with reasoning

1. **Split "front door" from "operator console" explicitly, and only rebuild the former.** The two-pane conversational-plus-workspace layout, the Approve/Edit/Skip triage cards, and the graduated-autonomy 90-percent/20-reps calibration rule from the shelved command-center design are the reusable assets; none of the current code is multi-tenant-shaped, so build the client-facing front door as a new, real, multi-tenant web app rather than trying to generalize any one of the four half-built GUI attempts. This closes scorecard gap #1 ("Build one real instance") directly.

2. **Consolidate, do not rebuild, the-sage's OKR scoring core and the fieldy-clawdia/recall-notetaker capture pipeline.** Both are validated on real data at meaningful scale (99.3 percent scoring accuracy; 343 notes and 47k embeddings). Wrap them behind a service boundary that adds per-tenant data isolation and per-tenant credentials, but do not rewrite the scoring or extraction logic itself. Fix the capture pipeline's known dedup bug (today's call landed three times, per the overview) before onboarding a second tenant, since duplication bugs compound with tenant count rather than staying constant.

3. **Treat the gamification engine as reusable math, not a reusable data model.** The points/rarity/tier formulas are real and live, but today's implementation is a single running tally for one person. The jump to per-person, per-org leaderboards with anti-abuse caps is a genuine rebuild of the data layer, and it is explicitly blocked on Nick's mechanics spec (due 7/31). Do not start that rebuild before the spec lands, to avoid building it twice.

4. **Resolve native-vs-web by scope, not by re-litigating the rule.** Keep the native operator console (claudecode-bar's lineage) for Jarred and future internal operators; ship the client-facing V1 product as a web app with standard SAML/OIDC redirect-based SSO. This is not a violation of the "native app" law so much as a correction of its scope: the law was proven at n=1 for a fleet-operator HUD, and there is no evidence anywhere in the estate that it was ever tested against an external, multi-tenant, SSO-gated buyer. If leadership still wants a native wrapper for brand reasons, prefer Tauri for a new build unless an enterprise buyer's IT team requires deep Windows device-compliance (Conditional Access/WAM), in which case Electron's more mature identity tooling is the safer choice. Either way, never do OAuth in an embedded webview; always shell to the system browser.

5. **Keep ClickUp as the default backbone, but build an adapter, not a hard dependency, from day one.** The sunk integration depth and the fact that outcome-based KR scoring is explicitly blocked on ClickUp's data today argue for continuity. But restrict all new, client-facing backbone code to ClickUp's public API plane; do not extend the internal-frontdoor JWT-harvest technique to any client-owned workspace, since the estate's own capability map already labels it ToS-gray and suspension-risk, scoped to a single owned workspace. Build the Linear adapter in parallel (not necessarily at V1 launch, but designed-for from the start), since Linear's superior API ergonomics and its status as a named MCP Enterprise-Managed Authorization adopter make it the more likely tool of record for a security-conscious client, or for clients who are already Linear shops and will not switch their engineering tracker for a vendor's convenience.

6. **Build the tool-connector and skill-marketplace layer on MCP, using its Enterprise-Managed Authorization extension for client tool access, and budget for the parts the SDK does not provide.** Do not build a bespoke auth system for ClickUp/Linear/Slack-class integrations; MCP's OAuth 2.1 flow already gives scoped, short-lived, multi-tenant-aware tokens for free. Do explicitly plan and staff for retry logic, per-tenant budget caps, durable execution, and observability, since the SDK does not ship these and the estate has independently arrived at the same gap list (pai-observatory, teamclaude-fleet). Price in the risk that Anthropic's paused programmatic-usage metering change could return; do not build V1 unit economics assuming today's subscription-covered token costs are permanent.

## Risks to this recommendation

- **The adapter-not-lock-in approach to ClickUp costs real engineering time up front** that a straight ClickUp-only build would not; if the 7/17-18 pitch or the 7/31 mechanics deadline compresses the timeline, there will be pressure to hardcode ClickUp calls directly into product logic "just for the pitch," which then has to be unwound later. Flag this explicitly before build starts.
- **All pricing figures for ClickUp Enterprise, Linear Enterprise, and Claude Enterprise are third-party estimates**, since none of the three vendors publish Enterprise per-seat pricing; actual contract terms could land meaningfully above or below the $20-30/user/month range cited across sources, and a real quote should be pulled before any unit-economics commitment is made in the deck.
- **The Tauri-vs-Electron and native-vs-web analysis is generic industry guidance, not validated against this specific buyer set** (Anchorage, Upexi, future enterprise clients); if a specific target client's IT security team has an unusual hard requirement (for example, mandatory Windows Conditional Access device compliance, which is common in regulated finance), that could flip the recommendation toward Electron or even toward keeping a native shell for the client product, not just the operator console. Confirm with Anchorage's security posture specifically, given its existing $25k/mo relationship and known compliance sophistication.
- **The MCP ecosystem's own security track record (30+ CVEs, the Asana and Smithery incidents) is a real and current caution**, not a settled solved problem; building the product's tool-connector layer on MCP should not be read as "auth is handled," only as "auth has a standard to build against." A dedicated security review of the product's own MCP server implementations should happen before any client's ClickUp/Linear/Slack credentials flow through it.
- **The "Fieldy/Recall capture pipeline is 85 percent built, do not rebuild" claim is Jarred's own assessment**, carried over from the concept overview; this dossier did not independently re-verify that estimate against the actual repo, and the known live dedup bug (today's call landed three times) suggests the multi-tenant productization gap in that pipeline may be larger than "15 percent."
- **This research is time-boxed web-and-local-scan work, not a formal build estimate.** Effort sizing (how many weeks each consolidation or rebuild actually takes) is out of scope here and should follow as a separate technical-planning pass, ideally with Carter, before the 7/15 concept doc commits to a specific build sequence.
